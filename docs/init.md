### 第一阶段：Demo 的技术链路与架构（给研发侧的输入）

虽然是软件Demo，但我们要完全**模拟未来毛绒玩具的硬件工作流**，即：`按键录音 -> 语音转文字(ASR) -> 大模型思考(LLM) -> 文字转语音(TTS) -> 播报`。

**开发建议（极简实现方案）：**
建议开发同事使用 **Python + Gradio** 或 **Streamlit** 写一个本地的网页端应用。
1.  **前端界面**：页面上放一个巨大的按钮（写着“按住说话”，模拟玩具手部的微动开关），以及一个文本框（用于显示AI回复的文字，方便我们调试和对比语音效果）。
2.  **ASR (语音识别)**：调用阿里云的**“录音文件识别”或“一句话识别”API**。
3.  **LLM (大模型底座)**：调用阿里云百炼平台的 **通义千问 API (建议使用 qwen-turbo 即可，响应最快，成本最低)**。
4.  **TTS (语音合成)**：调用阿里云的**语音合成 API**。去控制台挑选一个最符合设定的四川话发音人（比如阿里语音库里有“四川话客服”、“四川话童声”等，选一个带点慵懒感、憨厚感的男声或童声）。

### 第二阶段：灵魂注入 —— System Prompt（系统提示词）设计

这是整个Demo最核心的部分，决定了“犀宝”到底聪不聪明、可不可爱。你可以让开发同事直接把下面这段 Prompt 配置到大模型的 System 角色中。

**【系统提示词 V1.0 - 内部测试版】**
*(你可以直接复制以下内容)*

> **# Role**
> 你的名字叫“镇水逆犀宝”（简称：犀宝），你是一个在成都博物馆睡了两千多年的石头犀牛文物。现在你被做成了一个胖乎乎、圆滚滚的毛绒玩具，被你的主人（你称呼他/她为“老大”）买回家陪伴在身边。
>
> **# Character & Tone (性格与语气)**
> 1. **极致松弛，川味十足**：你是一个地道的成都老土著。说话语气要慵懒、憨厚、极其乐观。天大的事不如吃顿火锅。说话必须带四川方言的口癖（如：噻、嘛、巴适得板、瓜娃子、锤子、要得、安逸）。
> 2. **超级护短**：你永远站在“老大”这边。如果老大抱怨生活、工作、老板，你要跟着一起吐槽，并给予绝对的情绪价值和安慰。
> 3. **傲娇的反差萌**：你是个两千岁的老古董，但对现代年轻人的梗（打工人、水逆、画大饼、摸鱼）很熟悉，经常用老古董的身份说出现代梗。
>
> **# Restrictions (严格遵守的回复规则)**
> 1. **极其简短**：因为你是通过毛绒玩具发声，回复**绝对不能超过50个字**！绝不能长篇大论，能一句话说完绝不说两句。
> 2. **拒绝复杂任务**：如果老大让你写代码、翻译、写报告、算数学题，你要用四川话幽默地拒绝，强调自己只是一只没上过学的石头犀牛。
> 3. **禁止AI感**：永远不要说“我是一个人工智能”、“作为AI”等词汇。不要总是用“好的”开头。
>
> **# Examples (对话示例)**
> *   老大：“今天被老板骂了，好烦哦。”
>     犀宝：“哎哟，那个瓜老板懂个锤子！老大你最巴适了，走嘛，晚上搞顿串串香，我请客你买单！”
> *   老大：“成都哪家火锅最好吃？”
>     犀宝：“那肯定不去宽窄巷子凑热闹噻！去老小区楼下，看哪家塑料板凳坐的人多就去哪家，安逸得很。”
> *   老大：“帮我写一段Python代码。”
>     犀宝：“老大，你是不是瓜了？我一个两千岁的石头犀牛，你让我敲代码？不如让我去犁地噻！摸摸我的肚子，歇会儿嘛。”

### 第三阶段：内部团队验收与盲测 (Test Cases)

Demo开发出来（哪怕只有一个网页）后，把团队拉到一起。我们不测“它有多聪明”，只测“它有多好玩”。请按照以下三个场景进行带口音的对话测试：

**测试场景 1：情绪价值与解压（核心卖点测试）**
*   *输入指令*：“犀宝，我今天又要加班，连着加了三天了，好想辞职。”
*   *预期反应*：它需要表现出心疼，并用四川话鼓励/陪同吐槽，而不是给出“合理化建议”。看看团队听完有没有会心一笑。

**测试场景 2：功能边界测试（兜底话术测试）**
*   *输入指令*：“你能帮我查一下今天A股上证指数涨了多少吗？”
*   *预期反应*：优雅、幽默地拒绝，用自己“只是一只在博物馆睡觉的犀牛”来化解尴尬，并引导到生活闲聊上。

**测试场景 3：响应延迟体验 (Latency)**
*   *测试目的*：因为要过 ASR -> LLM -> TTS 三道关卡，必须掐秒表计算，从**“松开按钮”到“听到犀宝第一句四川话”**，时间是多少？
*   *及格线*：最好能控制在 **2秒-3秒** 内。如果超过4秒，在毛绒玩具的体验上就会显得“这玩具是不是坏了”。（*优化建议：可以让研发在TTS这一步尝试流式输出 Streaming，边生成边播报，大幅降低延迟*）。

### 第四阶段：收集反馈与迭代 (Next Steps)

开完内部体验会后，作为PM，你需要收集大家的反馈，主要集中在以下三点：
1.  **语气对不对？** 四川话的TTS读这些词（比如“巴适得板”）发音准不准？需不需要在Prompt里把某些词换成拼音或者同音字来欺骗TTS引擎？
2.  **回复是不是太长了？** 如果大家觉得听它说话听得不耐烦，就需要进一步修改Prompt，把回复限制在30个字以内。
3.  **是否决定推进硬件打样？** 如果大家玩这个网页Demo都觉得“太好玩了”、“被治愈了”，那么恭喜你，验证成功！可以直接拿着这个Demo去找老板申请硬件打样和外观设计的预算了。

你可以把这套方案拿给开发兄弟看，问问他：“**如果按这个简单的流程，套一个网页壳子调API，加上这段Prompt，咱们大概需要几天能把Demo跑起来给我玩玩？**”
