### 一、 后端核心框架选型：FastAPI

强烈建议使用 Python 的 **FastAPI** 框架。
*   **为什么选它？** 这是目前 Python 生态里做高并发、异步 I/O 最强的框架。我们未来要对接阿里云的 ASR、LLM、TTS，全都是网络 I/O 操作。FastAPI 原生支持异步（`async/await`）和 WebSocket，能把延迟压到最低。
*   **扩展性**：现在我们只跑一个Demo网页，未来有一万个“犀宝”同时在线，FastAPI 也能轻松承载。

### 二、 后端核心模块拆解 (Micro-architecture)

整个后端不需要搞得很复杂，但必须**模块化解耦**。建议开发同事将后端代码拆分成以下 4 个核心微模块：

#### 模块 1：接入与路由层 (Connection & Router Module)
*   **职责**：负责和前端（网页/小程序/硬件）建立连接。
*   **协议**：使用 **WebSocket** 建立长连接（而不是普通的 HTTP POST）。
*   **逻辑**：
    *   接收前端传来的音频流（或者音频文件）。
    *   向前端实时下发合成好的四川话音频流（Audio Stream）。
    *   *PM视角*：这就好比给犀宝修了一条高速公路，双向通车，不用每次说话都重新建路。

#### 模块 2：感知层 / ASR 引擎 (Speech-to-Text Module)
*   **职责**：把用户的语音转化为文字。
*   **对接**：调用阿里云 `qwen3-asr-flash-realtime`。
*   **优化点**：如果是硬件端，建议采用静音检测（VAD），用户说话停顿超过 0.8 秒，自动截断音频发给大模型，不用等用户松开按键，这样能抢出几百毫秒的时间。

#### 模块 3：认知层 / 记忆与大模型引擎 (LLM & Memory Module)
*   **职责**：犀宝的“大脑”，负责生成幽默的川味回复。
*   **对接**：调用阿里云 `qwen3.5-plus`。
*   **核心逻辑**：
    1.  **System Prompt 注入**：每次请求都带上我们写好的“两千岁成都犀牛”人设提示词。
    2.  **流式输出 (Streaming)**：必须开启 `stream=True`。大模型生成一个字，就吐出一个字，绝对不能等整句话写完。
    3.  **短时记忆 (Session Memory)**：玩具需要有记忆。后端需要在内存（或 Redis）里维护一个 `Session ID`。比如用户刚刚吐槽了老板，下一句说“他还让我加班”，犀宝得知道“他”指的是老板。设定保留最近 5-10 轮对话即可，过期自动清空（玩具休眠即失忆，符合毛绒玩具定位）。

#### 模块 4：表达层 / TTS 引擎 (Text-to-Speech Module)
*   **职责**：把文字变成 Eric 的四川话发音。
*   **对接**：调用阿里云 `qwen3-tts-instruct-flash-realtime`。
*   **高阶挑战（决定生死的延迟优化）**：
    *   这里有一个核心的工程挑战：TTS 引擎通常需要一整句话才能准确发音（因为涉及到语调和断句）。
    *   开发需要在这里做一个 **“句子截断器 (Chunker)”**：当 LLM 流式吐出文字时，后端代码要监控标点符号（如逗号、句号、感叹号、问号）。一旦遇到标点符号，立刻把这一小段文字切下来，扔给 TTS 去合成音频并直接发给前端播放。
    *   *效果*：当大模型还在想下半句的时候，犀宝的嘴巴已经开始播放上半句了。这是实现“秒回”的终极秘诀。

### 三、 数据流转全景图 (Data Flow)

你可以让开发同事脑海中形成这样一条流水线：

1. `[Client]` 发送录音字节流 ➔ `[Backend: WebSocket]`
2. `[Backend: ASR]` 将字节流推给阿里，拿到文字：*"我今天好累哦"* ➔ 传递给 `[Backend: LLM]`
3. `[Backend: LLM]` 结合历史记忆和 Prompt，向阿里请求回复，收到流式文字：*"哎"* ➔ *"哟"* ➔ *"，"* 
4. `[Backend: Chunking]` 遇到逗号，截断！把 *"哎哟，"* 推给 `[Backend: TTS]`
5. `[Backend: TTS]` 请求阿里拿到 *"哎哟，"* 的四川话音频 ➔ 推给 `[Backend: WebSocket]`
6. `[Client]` 瞬间开始播放音频：“哎哟...” 
7. *(与此同时，大模型还在生成后半句：“那个瓜老板又让你加班了嘛？”并重复上述过程)*

### 四、 PM 需要和开发确认的 3 个边界问题

既然重心转到了后端，有几个产品逻辑需要你现在拍板，开发才能写到代码里：

1.  **异常断网/超时兜底**：如果阿里云接口挂了，或者响应超过 5 秒，后端返回什么？
    *   *建议策略*：后端直接切断外部请求，播放一段本地预置的音频（比如：“哎呀，老大，我脑壳有点昏，让我睡一会儿嘛”）。
2.  **单次对话的最大长度**：为了防止熊孩子一直按着不松手，或者周围环境音一直录入导致大模型 Token 费用爆炸，我们需要限制单次录音时长吗？
    *   *建议策略*：后端强制设定单次音频最大接收时长为 **15 秒**，超过 15 秒强制截断并开始回答。
3.  **记忆重置周期**：玩具的“记忆”保持多久？
    *   *建议策略*：后端设定 Session 超时时间为 **10 分钟**。10 分钟内没有新对话，自动清除上下文记忆，进入新的对话周期。

把这些逻辑敲定后，后端的框架就犹如有了坚实的钢筋水泥。开发同事可以直接用 FastAPI 把这个管道搭起来了。
